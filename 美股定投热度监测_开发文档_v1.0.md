# 美股定投热度监测系统 · 开发文档(v1.0)

> 配套设计文档:《美股定投热度监测方案 v2.5(冻结版)》。设计文档回答"建什么",本文档回答"怎么建"。两份文档冲突时以设计文档为准,唯一例外:本文档第 4 节(防前视偏差规则)优先级最高——违反它的回测结果一律无效,无论设计文档怎么写。

---

## 0. 给 Claude Code 的执行说明

1. 将本文档与设计文档放入仓库根目录,并在 CLAUDE.md 中声明两者为长期上下文。
2. 按里程碑 M0 → M4 顺序开发,每个里程碑有验收标准,**通过后才能进入下一个**。
3. 设计文档已冻结:任何对权重、阈值、公式、窗口的"优化"想法,记 issue,不改代码。
4. 所有数值参数只允许存在于 `config.yaml`,代码中出现魔法数字视为缺陷。
5. 不确定时宁可实现得保守并留 TODO,不要擅自补全设计;数据拿不到时降级运行并在输出中显式标注缺失项,不要静默跳过。

## 3. config.yaml

全部数字从设计文档逐项抄录,分组:`weights`(维度间 40/25/25/10 与各维度内)、`windows`(估值滚动 180 月,其余 expanding)、`bands`(15/35/60/75/88 边界与各档动作)、`corrections`(修正一 +5、修正三 +3、修正三触发分位 90、新高判定 1%、广度阈值 55)、`sell`(88、85、20 日、月减 5%、上限 30%、衰退警报时 10 日)、`hysteresis`(10 日)、`anchors`(VIX 40、HY 8%、回撤 0.30/0.40)、`breadth_knots`(U 型五节点)。
交付物之一:`tests/test_config_matches_doc.py` 把 config 的每个数字与设计文档中的表格值做一次硬断言,防两处漂移。

## 4. 防前视偏差规则(最高优先级)

回放中,交易日 t 的热度只允许使用"t 日收盘后当时可得"的信息:

1. **分位数窗口截止 t**:扩展窗口 = 序列起点至 t;滚动窗口 = 截至 t 的最近 N 个观测。`pct()` 必须支持 `as_of` 切片,或由回放循环先切片再传入。
2. **CAPE 与估值类月度数据**:t 日使用"最近一个已发布完整月份"的值,发布滞后保守设为 1 个月(即 3 月的任意一天用 2 月值)。
3. **GDP**:发布滞后约 1 个季度,t 日使用 t−1 季度值;ffill 的方向必须是"向后填充旧值",M0 写断言验证。
4. **周线 RSI**:周中只使用截至 t 的**最后一个已完成周**;`resample("W-FRI")` 的当周未完成 bar 必须丢弃。
5. **cummax / rolling** 本身无前视,但注意整段序列预计算后再切片取值是允许的,前提是计算只依赖过去(cummax 满足;rolling 满足;任何 center=True 或未来窗口的操作禁止)。
6. **前视探测测试(M2 验收必含)**:随机抽 50 个历史日期 t,断言该日传入打分函数的所有序列的最大时间戳 ≤ t(月度数据 ≤ 上月末)。

## 6. 校准断言(设计文档 §5 的可执行翻译)

## 7. 单元测试规格(M1 验收,期望值已在设计阶段实测)

1. **回撤方向**:构造带漂移随机游走(seed=42,8000 日,日收益 N(0.0004, 0.01)),截至价格最大值日,回撤项得分应在 (85, 99.9);追加 150 日线性 −35% 崩盘后,得分应 < 10。并断言旧逻辑(负数口径 + invert)在同一底部给分 > 90,作为 bug 回归测试。
2. **mid-rank**:同序列新高日得分 ≈ 100 − 新高日占比/2,不得恒为 100。
3. **breadth_score**:9 个测试点 {0:0, 10:5, 20:10, 37.5:30, 55:50, 70:54, 85:58, 92.5:61.5, 100:65}。
4. **装配逻辑**:全指标 50 分时,(base, heat_spx, heat_ndx) 在四种修正组合下分别为 (50,50,50)、(50,55,50)、(50,50,58)、(50,55,58);全指标 99 分 + 全修正时两热度均封顶 100。
5. **迟滞与卖出状态机**:喂入构造的热度序列,断言:档位在阈值附近抖动 9 日不切换、第 10 日切换;sell_days 在任一条件断开当日清零;`load_state` 对旧版单热度结构返回 default。

## 10. 明确不做的事

不做参数优化/网格搜索;不做机器学习;不加新指标;**不接任何实盘交易 API——系统只提醒,下单永远由人执行**(这既是设计哲学也是安全边界);回测证据若提示设计需要修订,流程是开设计文档新版本号 + 附录续写,而不是直接改代码。
